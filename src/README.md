# AIサービス個人カタログ

増え続ける AI サービス / 技術を、契約数の増減ではなく「比較可能な能力資産」として管理する。

このカタログは、サービス一覧ではなく、黒澤用途での使い分け判断を育てるための台帳。

## 構成（番号の十の位＝層）

置き場に迷ったら、まず「これはどの層か」を決める。

| 層 | 意味 | 中身 |
|---|---|---|
| **10番台** | **判断** — 決めたこと | 契約・活用戦略・使い分けの正本 |
| **20番台** | **台帳** — 調べたこと | モデル・サービス・用途・候補の参照データ |
| **30番台** | **証拠** — 確かめたこと | 評価の計画とログ |

```text
10_契約/              01_現在の契約 → 02_バーベル戦略 → 03_将来方針（番号＝移行経路順）
11_活用戦略/          契約済みキーをどう使うか（Open WebUI / reasoning-orchestrator / ai-gateway）
12_使い分けルール.md   用途 × サービスの選択正本
13_教材連携.md        外部教材リポ ai-tool への外部化ルール

20_モデル/            プロバイダ × モデル階層。プラン比較
21_サービス/          サービス / 技術ごとの台帳（18件）
22_用途/              用途からの逆引き（8件）
23_候補/              未検証候補

30_評価/              計画 / ログ / テンプレート
```

### 層の依存方向

```text
30_評価（証拠）
  ↓ 根拠を与える
20番台（台帳）
  ↓ 材料を与える
10番台（判断）
```

**判断は台帳を根拠にし、台帳は証拠を根拠にする。** 逆流させない。
証拠のない判断は仮説として明示する。

## 運用ルール

- まず `23_候補/` に置き、必要なものだけ `21_サービス/` へ昇格する。
- `21_サービス/` に作っただけでは採用済み扱いにしない。評価ログがついて初めて判断材料になる。
- 用途で迷ったら `22_用途/` を見る。
- 使い分けが固まったら `12_使い分けルール.md` に反映する。
- 料金、利用規約、API、CLI、MCP、ローカル可否は変わるため、サービス台帳へ昇格する時点で確認する。
- 教材・比較記事として外に出す内容は `/Users/kurosawa/Dev/ai-tool` へ置く。ここには判断根拠と評価ログを残す。

## 今すぐ使う順番

1. 用途が決まっているなら `22_用途/` を見る。
2. 具体サービスの性質を見るなら `21_サービス/README.md` から個別台帳へ行く。

4. 実際に使ったら `30_評価/ログ/` にログを作る。
5. 判断が固まったら `12_使い分けルール.md` を更新する。

## 評価の型

```text
タスク
  -> 入力
  -> 使用サービス / 技術
  -> 結果
  -> 理由
  -> 再利用判断
  -> 12_使い分けルール.md へ反映
```

「Claude は賢い」「DeepSeek は安い」ではなく、
「認証レビューでは Claude が設計矛盾を拾ったので優先」のように残す。
