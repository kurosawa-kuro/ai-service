## ハーネス適性

- ワークフロー組み込み: 高候補
- 接続方式: API
- 制約: robots.txt、利用規約、クロール対象、コスト、抽出品質を確認する

## 強み

- Webページを LLM / RAG に入れやすい形へ抽出する候補。
- Tavily / Exa が「探す」寄りなら、Firecrawl は「取って整える」寄りとして評価する。
- AIサービス台帳の公式ページ収集にも使える可能性がある。

## 弱み

- クロール対象サイトの規約確認が必要。
- APIコストと抽出品質を実測しないと判断できない。
- 検索そのものより、抽出・整形の役割として分ける必要がある。

## 得意用途

- 公式docsの抽出。
- サービス情報の台帳化補助。
- RAG投入前のWeb資料整形。

## 苦手用途

- 調査結果の最終判断。
- 出典の信頼性評価。
- ログインが必要なページや権限が絡むページ。

## 黒澤用途

- AIサービス候補の公式ページ収集。
- 講座資料のWeb素材整理。
- 自作AIルーター / RAG の取り込み部品。
