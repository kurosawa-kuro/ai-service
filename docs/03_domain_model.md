# 03 ドメインモデル

## 用語

| 用語 | 意味 |
|---|---|
| AIサービス / 技術 | カタログ化する対象。商用サービス、OSS、API基盤、CLI、MCP、ローカル実行環境などを含む。 |
| 能力カテゴリ | サービス名ではなく能力で分類する正本の棚。`docs/02_architecture.md` の 16 分類を使う。 |
| サービス台帳 | `src/21_サービス/` 配下の 1 サービス / 技術ごとの記録。基本情報、ハーネス適性、強み、弱み、黒澤用途、検証ログ、継続評価を持つ。 |
| 用途逆引き | `src/22_用途/` 配下の用途別ページ。第一候補、低コスト補助、ローカル検証、判断ルール、破綻条件を持つ。 |
| 評価ログ | `src/30_評価/` 配下の実タスク結果。タスク、入力、使用サービス、結果、理由、再利用判断を残す。 |
| ルーティングルール | `src/12_使い分けルール.md`。用途からサービス / 技術を選ぶ判断の正本。 |
| ハーネス適性 | API / CLI / IDE / MCP / AGENTS.md / rules / hooks / ローカル実行など、自分のワークフローへ組み込める度合い。 |
| 黒澤用途 | 一般論ではなく、黒澤さんの実務、講座、PM、コーディング、調査、トレード材料整理などでの勝ち筋。 |
| 継続評価 | A / B / C などで表す継続利用判断。必ず理由と観察ポイントを添える。 |
| 候補棚 | `src/23_候補/` 配下の未検証候補。採用判断ではなく、後で台帳へ昇格するための素材。 |

## 状態 / ライフサイクル

```text
候補棚
  -> サービス台帳へ登録
  -> 用途逆引きへ関連付け
  -> 実タスクで評価ログを追加
  -> ルーティングルールへ判断を反映
  -> 月次レビューで継続評価 / 観察ポイントを更新
```

## 評価原則

- 「契約を増やす / 減らす」ではなく、増え続ける AI を比較可能な能力資産として管理する。
- サービス名の一覧だけで管理しない。能力カテゴリ、用途、ハーネス適性、評価ログで管理する。
- 価格表だけを作らない。無料枠、API、CLI、MCP、ローカル可否、実務での勝ち筋、再利用判断まで残す。
- 印象評価だけで更新しない。評価ログにはタスク、入力、結果、理由、再利用判断を残す。
- コーディング AI は一括りにしない。補完、チャット、差分修正、CLI実行、リポジトリ理解、Issue駆動、PR作成、レビュー、テスト生成、ドキュメント更新、MCP接続を分けて見る。

## 関連タスク

- 用語、状態、ライフサイクルの変更は task に背景と影響範囲を残してから反映する。
- 未確定の業務ルールはこの文書へ入れず、`docs/tasks/backlog/` で調査対象として管理する。
- 確定したドメイン判断は、必要なら `docs/adr/` にも残す。
