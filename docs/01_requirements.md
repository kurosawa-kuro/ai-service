# 01 要件

## 目的

増え続ける AI サービスを「どれを契約するか」の単発判断で消耗するのではなく、
比較可能な能力資産（黒澤専用の能力地図 / 武器庫）として管理する個人カタログを作る。

- サービス名ベースではなく、能力カテゴリ・用途・ハーネス適性・評価ログで管理する。
- 公式スペックの転記より「黒澤用途でどうだったか（勝ち筋）」を残すことを優先する。
- 最初から完璧な Web アプリ / DB を作らず、Markdown + Git + 月次レビューで始めて段階的に育てる。

## 範囲

対象:

- AI サービス / 技術のカタログ化（分類・基本スペック・無料枠・API/CLI/MCP/ローカル可否・ハーネス適性）。
- 用途からの逆引きインデックスと使い分けルール（routing-rules）。
- 黒澤用途での評価ログ蓄積（タスク・入力・結果・理由・再利用判断）。
- Markdown 台帳から始め、用途別 index → 評価ログ → Supabase → 検索 → 自動ルーティングへ段階的に成長させる。

非対象:

- 「ChatGPT と Claude どっちがいい」式の単発・感想ベースの採用判断。
- 最初から完璧な Web アプリ / DB 化（早すぎる作り込み）。
- 公式スペックの網羅的転記そのものを目的にすること。
- 全候補サービスの即時契約・即時評価（候補の棚出しは素材であって採用ではない）。

## ユーザー

- 黒澤さん本人（個人）。カタログの作成者・利用者・評価者を兼ねる。

## ユースケース

| ID | ユースケース | 成功条件 |
|---|---|---|
| UC-001 | 新しい AI サービスを台帳に追加する | 固定フォーマット（分類・スペック・ハーネス適性・黒澤用途・評価ログ）で 1 ファイル追記できる |
| UC-002 | 用途から逆引きして使い分ける | 用途ページで第一候補 / 低コスト補助 / ローカル検証 / 判断ルール / 破綻条件を引ける |
| UC-003 | 実タスクの評価ログを残す | タスク・入力・結果・理由・再利用判断が記録され、次回の選択根拠に使える |
| UC-004 | 月次レビューでカタログを更新する | 継続評価（A/B/C 等）と観察ポイントが更新され、古い印象評価が上書きされる |
| UC-005 | ハーネス接続可否で判断する | API/CLI/MCP/AGENTS.md/ローカル可否から、自分のワークフローに組み込めるか判断できる |

## 制約・前提

- サービスは今後 10 → 50 → 100 個に増える前提。増えても破綻しない分類軸（能力カテゴリ）を正本にする。
- 評価は印象（「賢い」「安い」）でなく再現可能なログで残す。
- 機密に近い内容はローカル実行または契約上許可された環境に限定する。
