# 05 データモデル

## 設定

| 種別 | 方針 |
|---|---|
| 一般設定 | `env/config.yaml` または環境変数 |
| ローカル秘密情報 | ignore するローカルファイル |
| 共有 / 本番の秘密情報 | Doppler などの secret manager |

## 永続化

Phase 1 では Markdown + Git を永続化方式とする。

```text
catalog/
  ├── services/
  ├── usecases/
  ├── evaluations/
  ├── candidates/
  └── routing-rules.md
```

DB 化は構造が固まってから検討する。

## Markdown レコード

### サービス台帳

担当パス: `catalog/services/*.md`

| 項目 | 方針 |
|---|---|
| サービス名 | 見出しに置く |
| 提供会社 | 基本情報に置く |
| 能力カテゴリ | `docs/02_architecture.md` の 16 分類から選ぶ |
| 種別 | 無料 / 有料 / ローカル / API / CLI など |
| 月額 | 未確認なら `TODO` |
| 無料枠 | 未確認なら `TODO` |
| API / CLI / IDE / MCP | 可 / 不可 / 未確認 |
| AGENTS.md / rules / hooks 対応 | 可 / 不可 / 未確認 |
| ローカル実行可否 | 可 / 不可 / 未確認 |
| 得意用途 / 苦手用途 | 黒澤用途に引き寄せて書く |
| ハーネス適性 | ワークフロー組み込み、接続方式、制約 |
| 検証ログ | 日付、タスク、入力、結果、理由、再利用判断 |
| 継続評価 | A / B / C、理由、観察ポイント |

### 用途逆引き

担当パス: `catalog/usecases/*.md`

| セクション | 内容 |
|---|---|
| 第一候補 | その用途で最初に試すサービス / 技術 |
| 低コスト補助 | 安価・無料枠・軽量処理向け |
| ローカル検証 | 機密、低コスト、大量処理、検証向け |
| 判断ルール | どういう条件でどれを選ぶか |
| 破綻条件 | その用途で避けるべき使い方 |
| 関連評価ログ | 根拠となる `catalog/evaluations/` への参照 |

### 評価ログ

担当パス: `catalog/evaluations/YYYY-MM-<topic>.md`

| 項目 | 内容 |
|---|---|
| タスク | 実施した作業 |
| 入力 | モデル / サービスへ渡した前提や素材 |
| 使用サービス / 技術 | 比較対象を含める |
| 結果 | 出力や観察結果 |
| 理由 | なぜその評価にしたか |
| 再利用判断 | 次回の用途選択にどう使うか |
| 次回へのメモ | 未検証点、観察ポイント |

### 候補棚

担当パス: `catalog/candidates/*.md`

未検証の AI サービス / 技術を置く。ここにあるだけでは採用済み、契約対象、推奨対象を意味しない。
サービス台帳へ昇格する前に、最新情報、利用規約、料金、API / CLI / MCP、ローカル可否を確認する。

## 関連タスク

- schema、migration、設定、永続化方式の変更は task に目的・移行手順・検証方法を残す。

- 確定した migration 手順は `docs/runbooks/` または `08_release_runbook.md` へ昇格する。
